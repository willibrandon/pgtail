# Implementation Plan: Tail Mode Command History & Autocomplete

**Branch**: `024-tail-history-completion` | **Date**: 2026-03-01 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/024-tail-history-completion/spec.md`

## Summary

Add command history with Up/Down arrow recall and context-aware ghost text autocomplete to the tail mode TailInput widget. Three new modules implement the core functionality: `TailCommandHistory` for persistent history with a three-state cursor model, `TailCommandSuggester` extending Textual's `Suggester` for structural completions with history fallback, and `tail_completion_data` defining per-command completion specs for all 20+ tail mode commands. Two existing modules are modified: `TailInput` gains history navigation key bindings and a synchronous guard pattern, and `TailApp` wires the new components during initialization and submission. A refactoring extraction from `tail_textual.py` addresses a pre-existing constitution violation (>900 LOC).

## Design Decisions

### D-001: Textual Suggester with Disabled Caching

Subclass `Suggester(use_cache=False, case_sensitive=True)`. Caching disabled because history changes on every submission (FR-020). Case-sensitive mode gives raw input for mixed-case handling (FR-018). See [research.md](research.md) R-001.

### D-002: Synchronous Boolean Guard Pattern

Use `_navigating: bool` instance flag on TailInput. Set `True` before programmatic value changes during navigation, cleared after. Textual's reactive watchers fire synchronously, making this reliable. See [research.md](research.md) R-002.

### D-003: Dataclass-Based Completion Specs

`CompletionSpec` frozen dataclass with optional fields (`static_values`, `dynamic_source`, `subcommands`, `flags`, `positionals`, `no_args`). Module-level `TAIL_COMPLETION_DATA` dictionary maps command names to specs. See [research.md](research.md) R-005.

### D-004: Dynamic Sources via Callables

Dictionary of `str -> Callable[[], list[str]]` passed from TailApp to TailCommandSuggester. Callables close over app state for lazy resolution. Three sources: highlighter_names, setting_keys, help_topics. See [research.md](research.md) R-007.

### D-005: Command Handler Extraction

Extract `_handle_command()` (lines 960-1133, ~173 LOC) and `_handle_export_command()` (lines 787-874, ~87 LOC) from `tail_textual.py` into new `tail_command_handler.py`. Both methods must be extracted to bring tail_textual.py from ~1145 to ~885 LOC (under the 900 LOC constitution limit). See [research.md](research.md) R-011.

### D-006: History File Path Duplication

New `get_tail_history_path()` function in `tail_history.py` duplicates the platform-resolution pattern from `config.py`'s `get_history_path()` with different filename. Intentional duplication over API generalization. See [research.md](research.md) R-003.

### D-007: navigate_forward Return Tuple

`navigate_forward()` returns `tuple[str | None, bool]` where the boolean indicates whether saved input was restored (past-newest transition). This enables the caller (TailInput) to distinguish between "moved to another history entry" and "restored your original text" — both require the guard, but the distinction aids debugging and testing.

## Implementation Steps

### Step 1: TailCommandHistory (`tail_history.py`)

**New file**: `pgtail_py/tail_history.py` (~250 LOC)
**Test file**: `tests/test_tail_history.py` (~400 LOC)
**Dependencies**: None (stdlib only: pathlib, logging, os, sys)
**Spec coverage**: FR-001 through FR-009, FR-016 (search_prefix), FR-018 (case-sensitive search)

Implement the three-state cursor model, file persistence (load/save/compact), consecutive deduplication, entry limits, safety limits on load, and prefix search. All file I/O silently handles errors with DEBUG logging.

**Test matrix** (~45 tests):
- Navigation state transitions: all 7 transitions from data-model.md state table
- Edge cases: empty history, single entry, at-oldest Up, at-rest Down
- Add: empty/whitespace rejection, consecutive dedup, max_entries trimming
- File I/O: load (normal, empty, missing, corrupt, oversized lines, non-UTF-8), save (append, create dir), compact (threshold, error recovery)
- search_prefix: exact match excluded, case sensitivity, most-recent-first, no match

### Step 2: Completion Data (`tail_completion_data.py`)

**New file**: `pgtail_py/tail_completion_data.py` (~350 LOC)
**Test file**: `tests/test_tail_completion_data.py` (~200 LOC)
**Dependencies**: error_stats.py (SQLSTATE codes), cli_tail_help.py (COMMAND_HELP keys)
**Spec coverage**: FR-012, FR-013, FR-014 (flag type definitions), FR-015 (dynamic source keys), FR-019, FR-022

Define `CompletionSpec` dataclass and `TAIL_COMPLETION_DATA` dictionary with specs for all 20+ tail mode commands. Includes static value lists (LEVEL_VALUES, TIME_PRESETS, THRESHOLD_PRESETS, FORMAT_VALUES, SQLSTATE_CODES), dynamic source key references, flag maps with boolean vs value-taking distinction, positional sequences with free-form markers, subcommand trees (highlight, theme), and no-argument markers.

**Test matrix** (~15 tests):
- Every tail mode command has a spec entry (inventory test)
- Flag type assertions (boolean flags map to None, value-taking to CompletionSpec)
- Positional count matches command signatures
- Free-form positions are None in positional list
- No-argument commands have no_args=True
- Dynamic source keys exist in expected set
- SQLSTATE codes list is non-empty and well-formed

### Step 3: TailCommandSuggester (`tail_suggester.py`)

**New file**: `pgtail_py/tail_suggester.py` (~200 LOC)
**Test file**: `tests/test_tail_suggester.py` (~350 LOC)
**Dependencies**: tail_history.py (Step 1), tail_completion_data.py (Step 2), textual.suggester.Suggester
**Spec coverage**: FR-010, FR-011, FR-014 (flag scanning), FR-016 (history fallback), FR-017 (full-line return), FR-018 (case handling), FR-020 (no caching), FR-021 (input parsing), FR-022 (free-form skip)

Implement the suggestion pipeline: input parsing (str.split, trailing-space detection, --flag=value splitting), command name completion (case-insensitive), structural argument completion (composition rules, flag scanning algorithm), history prefix fallback, and full-line return assembly.

**Test matrix** (~40 tests):
- Input parsing: empty, single word, command+space, command+partial, trailing space, multiple spaces, --flag=value splitting
- Command name: case-insensitive prefix, first alphabetical match, exact match (empty suffix), no match
- Argument completion per structural type: static values, subcommands, flags (boolean skip, value-taking consumed/unconsumed), positionals, free-form skip
- Composition rules: partial starts with -- (flag mode), positional index tracking, exhausted positionals
- Flag scanning: --since consumed by 5m, --trend boolean never consumes, --code unconsumed, --flag=value always consumed
- History fallback: structural empty suffix triggers fallback, no structural match triggers fallback, history case-sensitive
- Full-line return: returned string starts with input value, suggestion appended
- Edge cases: no-argument commands, unknown commands, alias commands (p, f, q)

### Step 4: TailInput Modifications (`tail_input.py`)

**Modified file**: `pgtail_py/tail_input.py` (~150 LOC, up from 84)
**Test file**: `tests/test_tail_input_history.py` (~250 LOC)
**Dependencies**: tail_history.py (Step 1), textual.suggester.Suggester
**Spec coverage**: FR-002, FR-003, FR-004, FR-023 (constructor), FR-025 (guard), FR-026 (ghost acceptance reset)

Extend constructor to accept optional `history` and `suggester` parameters (backward compatible per FR-023). Add Up/Down key bindings. Implement `_navigating` guard pattern with `try/finally`. Add `watch_value` to reset navigation on non-guarded changes. Pass `suggester` to parent `Input.__init__()`.

**Test matrix** (~16 tests):
- Constructor: backward compatible (no history, no suggester), with history only, with both
- Up/Down: navigation updates value, cursor moves to end of text
- Guard: programmatic value change doesn't reset navigation, user typing does reset
- Ghost acceptance: Right arrow at end with suggestion resets navigation (FR-026)
- Edge cases: Up with no history (no-op), Down at rest (no-op), Escape clears and resets
- Focus: history navigation only when input has focus
- Existing bindings unaffected: q (quit if empty), Escape (clear and blur)

### Step 5: TailApp Wiring (`tail_textual.py`)

**Modified file**: `pgtail_py/tail_textual.py` (~10 new LOC of wiring)
**Dependencies**: Steps 1-4 (all components ready)
**Spec coverage**: FR-024 (wiring), FR-006 (load/compact on mount)

Add imports for TailCommandHistory, TailCommandSuggester, get_tail_history_path, TAIL_COMPLETION_DATA. Create history and suggester in `__init__` (or early in `compose`). Create dynamic_sources dict closing over app state. Pass to TailInput in `compose()`. Load and compact history in `on_mount()`. Add command to history and save in `on_input_submitted()`.

Integration verified via the Step 4 tests (Textual app tests with `pilot`).

### Step 6: Extract Command Handler (`tail_command_handler.py`)

**New file**: `pgtail_py/tail_command_handler.py` (~260 LOC extracted)
**Modified file**: `pgtail_py/tail_textual.py` (reduced from ~1145 to ~885 LOC)
**Dependencies**: None (pure refactoring)
**Spec coverage**: Constitution compliance (900 LOC file size limit)

Extract `_handle_command()` method (~173 LOC) and `_handle_export_command()` method (~87 LOC) from `TailApp` into a standalone module with top-level `handle_command()` and `handle_export_command()` functions. Both functions receive the necessary context (status, state, tailer, log_widget, callbacks) as parameters. TailApp delegates to the new functions, keeping only the call sites. Both extractions are required to bring tail_textual.py under the 900 LOC constitution limit.

**No new tests**: Existing tail mode tests cover command handling. Verify all existing tests pass after extraction.
